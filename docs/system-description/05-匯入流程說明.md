# 5. 匯入流程說明

## 主要依據檔案

- `app/blueprints/tasks/task_routes.py`
- `app/blueprints/tasks/nas_routes.py`
- `app/services/task_service.py`
- `app/services/execution_service.py`
- `app/jobs/executor.py`
- `app/services/mapping_metadata_service.py`

## 完整流程

依程式碼推測，任務檔案匯入流程如下：

1. 使用者於 `/tasks` 指定 NAS 根目錄索引與相對路徑。
2. `nas_service.resolve_nas_path` 驗證路徑格式、允許根目錄與實體存在性。
3. 系統建立任務 ID、任務資料夾、`files/`、`output/` 與 `meta.json`。
4. `task_service.record_task_in_db` 新增 `tasks` 資料表紀錄。
5. `enqueue_task_source_sync_job` 建立 `task_source_sync` 背景工作。
6. worker 將來源資料同步至 `task_store/<task_id>/files/`。
7. 同步過程更新 `meta.json` 中的來源同步狀態。
8. 同步完成後任務可進入流程執行、Mapping 檢查或來源比對。

## 暫存與寫入

任務建立初期先寫入 `meta.json` 與資料庫任務紀錄，再建立背景工作。若資料庫紀錄建立失敗，系統會刪除已建立的任務資料夾；若背景工作建立失敗，系統會刪除任務紀錄與任務資料夾，避免留下不完整任務。

## 處理結果

匯入結果主要呈現在任務列表與任務 metadata。檔案同步完成後，流程與 Mapping 模組可讀取 `files/` 內的來源檔。任務檔案版本變動會影響 Mapping 方案狀態，相關版本號記錄於 `task_file_states.files_revision`。

## 失敗復原與一致性

任務建立路由對關鍵步驟採用補償式處理。若任務資料庫紀錄寫入失敗，系統會刪除已建立的任務資料夾；若來源同步 job 建立失敗，系統會刪除任務紀錄並移除資料夾。來源同步執行中或完成後若發生部分檔案異常，完整補償規則需由 `task_source_sync` worker 實作確認，因此本文保留「此處需補充」。

## 與後續模組的銜接

檔案匯入完成後，任務 `files/` 成為所有流程與 Mapping 的來源 root。流程步驟中的檔案選擇、模板解析、Mapping 來源解析、來源比對預覽、標準對應檔案檢查，均以此 root 或任務 output root 為邊界。若使用者重新同步 NAS 或在任務檔案 API 中修改檔案，Mapping metadata 會透過檔案 revision 判斷既有方案是否需要重新檢查。

## 待補充項目

- 此處需補充 `task_source_sync` worker 內部複製規則的完整設計說明。
- 此處需補充同步失敗時是否保留部分檔案與重試策略。
